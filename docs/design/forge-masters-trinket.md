## 1. 🔴 Grounding: "adrenaline" IS a live vanilla trinket mechanic (phantom-API risk CLOSED)

The card's framing — *"Vanilla Valheim has no native adrenaline system… the
adrenaline-cap hook is UNVERIFIED and likely wrong"* — **is itself wrong, and the
inversion is load-bearing for the whole design.** Adrenaline is a real, fully-wired
vanilla mechanic (network v36 / Ashlands era) and it is **purpose-built for
Trinkets**. This is NOT a custom meter we have to invent; it is a vanilla system we
parameterize and hook. Both the wiki corpus and the decomp confirm it.

> Decomp paths below cite `assembly_valheim.decompiled.cs` (the local decomp at
> `~/valheim/worldgen-spike/decomp/`). Line numbers are from the authoring-box
> decomp on 2026-06-18 and may drift ±a few lines across game patches — the
> **member names are the durable anchor**, verify those against
> `assembly_valheim.dll` metadata if a line ref looks off (ADR-0001: vanilla is
> fair game to read and cite).

### 1.1 What the wiki says (corpus: `wiki/fandom/Adrenaline.md`)

- *"Adrenaline is a personal resource used by **Trinkets**. It is represented by an
  orange bar in the HUD."*
- *"Adrenaline is generated by player actions **when the player has a trinket
  equipped**. … The maximum amount of adrenaline is **defined by the equipped
  trinket**."*
- *"When the maximum amount of adrenaline is reached, **all the adrenaline is
  consumed and the trinket power is activated**. Adrenaline can build up again
  without delay, even while the trinket power is active."*
- Decay: *"slowly decreases after a short delay… delay 10→6 s as the bar fills;
  decrease rate −1→−4 per second, depleting faster the fuller the bar."*

That is *exactly* this mod's loop, with our power = "+5 durability to equipped gear"
substituted for a combat buff.

### 1.2 What the decomp says (the load-bearing members)

**The per-trinket cap (this is Daniel's "~80"):**
- `ItemDrop.ItemData.SharedData.m_maxAdrenaline` — `:57844`, under
  `[Header("Adrenaline")]` (`:57843`). A per-item float. **The equipped trinket sets
  the cap.** Setting this on our trinket's SharedData to `80f` *is* the "~80".
- It is summed into the player's live cap through the equipment-modifier system:
  `Humanoid.GetEquipmentMaxAdrenaline()` → `GetEquipmentModifier(10)` (`:21926`),
  where index **10** maps to `"m_maxAdrenaline"` in the `s_equipmentModifierSources`
  table (`:15427-15431`). So an equipped trinket's `m_maxAdrenaline` becomes the
  player's effective cap.

**The cap-fire hook (the thing we attach our effect to):**
- `ItemDrop.ItemData.SharedData.m_fullAdrenalineSE` — `:57846`. A **vanilla
  StatusEffect field on the item.** This is the trinket's "power."
- `Player.AddAdrenaline(float v)` — `:19475`. When adrenaline reaches cap, this
  method (`:19490-19514`) walks the player's **equipped** items, and for the item
  whose `m_fullAdrenalineSE != null` it applies that StatusEffect via
  `SEMan.AddStatusEffect`, then **resets adrenaline to 0** and spawns
  `m_adrenalinePopEffects`. Verbatim shape:
  ```csharp
  if (m_adrenaline >= maxAdrenaline && maxAdrenaline > 0f) {            // :19490
      foreach (ItemDrop.ItemData item in GetInventory().GetAllItems())  // :19492-19494
          if (item.m_equipped && item.m_shared.m_fullAdrenalineSE != null) {   // :19496
              flag = true;
              var se = GetSEMan().GetStatusEffect(item.m_shared.m_fullAdrenalineSE.NameHash());
              if (se != null) se.ResetTime();                           // :19502 already active → refresh
              else GetSEMan().AddStatusEffect(item.m_shared.m_fullAdrenalineSE); // :19506 apply
          }
      m_adrenaline = (flag ? 0f : maxAdrenaline);                       // :19510 consume on fire
      if (flag) m_adrenalinePopEffects.Create(transform.position, ...); // :19513 pop VFX
  }
  ```
- **This is why the design has almost no custom code on the trigger side.** We do
  NOT patch `AddAdrenaline`. We do NOT meter combat. Vanilla already: fills the bar
  from combat, decays it, detects the cap, fires the SE, and resets. Our only job is
  to **author the StatusEffect** that `m_fullAdrenalineSE` points at, and make that
  SE repair equipped durability.

**The trinket slot itself:**
- `ItemDrop.ItemData.ItemType.Trinket = 24` (`:57652`).
- The player has a dedicated single trinket slot: `Humanoid.m_trinketItem`
  (`:12876`), equipped through the `EquipItem` Trinket branch and surfaced visually
  via `VisEquipment.SetTrinketItem` (`:14159`). **Vanilla allows exactly ONE trinket
  equipped at a time** — see the design tension in §1.4.

**Combat build-up sources (vanilla, for tuning context — we don't implement these):**
`SharedData.m_attackAdrenaline` (`:884`, default `1`), `m_blockAdrenaline`
(`:57848`, `2`), `m_perfectBlockAdrenaline` (`:57850`, `5`), plus Player-level
`m_attackMissAdrenaline` (`:15526`, `−5`), `m_nonBlockDamageAdrenaline` (`:15528`,
`−5`), `m_staggerEnemyAdrenaline` (`:15530`, `5`). Global multiplier
`Game.m_adrenalineRate` (`:84056`, `1`). Decay curves `m_adrenalineDegen` /
`m_adrenalineDegenDelay` (`:15518/:15520`) drive the 10→6 s delay / −1→−4 /s decay
the wiki describes. **None of these are ours to build — they are the vanilla fill/
decay we inherit for free.**

### 1.3 Verdict — NOT a custom meter

The mechanic is **~95% vanilla.** The phantom-API risk the card flagged does not
exist: every member above is real and present in the shipped `assembly_valheim`.
The **only** custom code is:
1. one **StatusEffect** subclass that, on apply (cap-fire), repairs +5 durability on
   the player's equipped gear, and
2. the **item** registration that sets `m_maxAdrenaline = 80` and wires
   `m_fullAdrenalineSE` to (1).

No `AddAdrenaline` patch. No combat metering. No `UpdateEquipment` patch. This is a
*much* smaller build than the Sunstone Lens (which DID need a Harmony prefix on
`DrainEquipedItemDurability`).

### 1.4 ⚠️ Surfaced design tension (NOT a blocker — Daniel decides)

Vanilla allows **only one trinket equipped at a time** (`m_trinketItem` is a single
slot, `:12876`). SBPR **already ships a trinket** — the **Sunstone Lens**. Therefore
the Forge Master's Trinket and the Sunstone Lens are **mutually exclusive**: a player
cannot wear both, and any future Iron-Compass-as-trinket competes for the same slot.

This is a genuine content-design choice, not a technical defect. Options for Daniel
(does not block the spec):
- **(T1) Accept the contention** — trinkets are deliberately scarce/competing; the
  player picks one exploration boon per outing. (Vanilla's own design — bronze-health
  vs iron-stamina trinkets compete identically.)
- **(T2) Re-home one of them off the trinket slot** — e.g. move the new durability
  item to the **Utility** slot (belt slot, `m_utilityItem`, `:13194`). But the
  Utility slot is taken by the **Cartographer's Kit**, and crucially the *adrenaline
  cap-fire hook only fires for the equipped item carrying `m_fullAdrenalineSE`* — and
  adrenaline-as-a-system is **trinket-bound** in vanilla. Moving off the trinket slot
  means **abandoning the adrenaline mechanic entirely** and re-deriving a custom
  meter (the very thing §1.3 says we don't need). **Not recommended.**
- **(T3) Defer** — ship it on the trinket slot now, revisit multi-trinket support
  later (a multi-slot system would be a much larger, separate mod).

**Recommendation: T1.** The adrenaline mechanic is trinket-bound by vanilla design;
fighting that means throwing away the clean hook. Ship on the trinket slot, accept
the one-trinket-at-a-time scarcity as intended content design, flag it in the
player-facing description so it isn't a surprise.

## 2. Mechanic spec (build-ready)

### 2.1 The trigger — vanilla, parameterized

| Parameter | Value | Source / rationale |
|---|---|---|
| Fill source | vanilla combat adrenaline | We set nothing; vanilla fills on hit/block/parry/dodge/stagger (§1.2). |
| Cap (`m_maxAdrenaline`) | **80f** (config-exposed) | Daniel's "~80". Set on the trinket's SharedData. Exposed as a BepInEx config so Daniel can tune without a rebuild (Sunstone Lens precedent). |
| Cap detection | vanilla `Player.AddAdrenaline` `:19490` | Not ours. Fires when `m_adrenaline >= maxAdrenaline`. |
| On-fire action | apply our `m_fullAdrenalineSE` | The SE is the "trinket power" (§2.2). |
| Post-fire reset | vanilla sets `m_adrenaline = 0` `:19510` | Not ours. Bar empties; refills from continued combat. |

### 2.2 The effect — +5 durability to EQUIPPED gear (scope LOCKED 2026-06-17)

When the SE applies (one apply == one cap-fire), it repairs durability on the
player's **currently equipped** items and **only** those. This is the locked
equipped-only scope — explicitly NOT every durability item in the backpack.

**"Equipped gear" is defined by vanilla's own equipped-slot enumeration**, not by us
re-deriving it. Vanilla's `Humanoid.UpdateEquipment` (`:13162-13202`) drains
durability on exactly these slots each tick:

```
m_rightItem, m_leftItem, m_chestItem, m_legItem,
m_helmetItem, m_shoulderItem, m_utilityItem, m_trinketItem   // :13170-13201
```

Our repair targets **the same set**, reached from the clean side via the **public**
`Inventory.GetEquippedItems()` (`:57192` — returns every `item.m_equipped == true`),
filtered to `item.m_shared.m_useDurability == true`. Using the public API (not the
`protected`/`private` slot fields) is the same clean-side discipline the Sunstone
Lens uses for `GetEquippedLens` (`SunstoneLens.cs:314`). The trinket itself has
`m_useDurability == false` (§3.2), so it self-excludes naturally.

**The repair primitive (all vanilla, public):**
- Read max: `ItemData.GetMaxDurability()` (`:58130`) = `m_shared.m_maxDurability +
  (quality-1)·m_durabilityPerLevel` (`:58135`).
- `m_durability` is a **public float field** on `ItemData` (`:58116`-region; written
  directly by vanilla all over, e.g. `:13229`, `:22082`).
- Repair = `item.m_durability = Mathf.Min(item.m_durability + 5f, item.GetMaxDurability())`.
  Clamp at max so we never over-heal.

| Parameter | Value | Rationale |
|---|---|---|
| Amount | **+5 durability units** (config) | Daniel's spec. Flat absolute, NOT percentage. Config-exposed for tuning. |
| Targets | equipped items with `m_useDurability` | The 8 vanilla equip slots, via `GetEquippedItems()`. Equipped-only LOCK. |
| Per-target | +5 to **each** equipped piece | See §2.3 open question O1 — "+5 to each" vs "+5 to the most-worn one". |
| Clamp | `Min(cur+5, GetMaxDurability())` | Never exceed max. A fully-repaired piece is a no-op. |
| Application model | **one-shot on apply** | One cap-fire == one +5. The SE does its work in `Setup()`/apply, not per-tick. See §2.3 O2. |
| Cooldown / reset | none of ours | Vanilla's adrenaline reset-to-0 + refill *is* the natural cadence. Re-fill to 80 → fire again. |
| Interaction w/ vanilla durability loss | additive, independent | Gear still wears normally; this periodically tops it back up. Net effect: combat-heavy play self-maintains gear. |

### 2.3 Open mechanic questions (Daniel/architect refine — do not block scaffold)

- **O1 — "+5 to what, exactly?"** Three readable interpretations of Daniel's
  "+5 durability to gear":
  - **(O1a) +5 to every equipped durability piece** — most generous; one cap-fire
    tops up the whole kit. *Architect lean:* this best matches the "forge master
    keeps your whole loadout maintained" fantasy and is the simplest to reason about.
  - **(O1b) +5 to the single most-worn equipped piece** — most "thematic upkeep",
    rewards targeting; smaller power budget.
  - **(O1c) +5 split/shared across equipped pieces** — fiddly, weakest fantasy.
  - **Recommendation: O1a**, config flag to fall back to O1b if Daniel wants it
    tighter. (The impl reads the same `GetEquippedItems()` list either way; O1a vs
    O1b is just "all" vs "argmin durability%".)
- **O2 — application model.** One-shot-on-apply (recommended) vs a short
  while-active HoT. One-shot is simpler, matches the vanilla "power fires once per
  cap" semantics, and avoids a per-tick `UpdateStatusEffect` loop. A near-zero TTL SE
  (`m_ttl` small) that does its repair in `Setup()` and expires is the cleanest shape
  — vanilla `AddStatusEffect` **clones** the template (`:24397`) so the singleton SE
  asset is never mutated.
- **O3 — "weapon in hand" emphasis?** The card asks whether it's "all equipped
  pieces, the weapon in hand, or both." Recommendation: **all equipped pieces (O1a)**;
  the in-hand weapon is just `m_rightItem`/`m_leftItem`, already in the set. No
  special-casing unless Daniel wants a weapon-only variant.
- **O4 — cap value.** 80 is Daniel's "~80". Vanilla bronze/iron trinkets ship
  different caps; 80 sits mid-range. Config-exposed; final number is a playtest knob.

## 3. Item spec + vanilla hooks (build-ready)

### 3.1 Slot — Trinket (ItemType.Trinket = 24)

The item goes in the **Trinket** slot (`ItemDrop.ItemData.ItemType.Trinket = 24`,
`:57652`; player slot `m_trinketItem`, `:12876`). This is non-negotiable: the
adrenaline cap-fire hook (`m_fullAdrenalineSE`) is read off equipped items and the
whole adrenaline system is trinket-bound in vanilla. The Sunstone Lens already proves
this slot works for an SBPR additive item. (Slot contention with the Lens → §1.4.)

### 3.2 Item registration (additive, ADR-0006 — the Sunstone Lens pattern)

Build the prefab **additively** via the established `Assets.ConstructItemShell` path
(fresh `ZNetView` + `ItemDrop` + `SharedData`), exactly like
`SunstoneLens.RegisterLensTrinket`. NO runtime prefab cloning (ADR-0006). Set on
`drop.m_itemData.m_shared`:

| Field | Value | Note |
|---|---|---|
| `m_name` / `m_description` | "Forge Master's Trinket" / flavor | Description should flag the one-trinket-slot tradeoff (§1.4) so it isn't a surprise. |
| `m_itemType` | `ItemType.Trinket` (24) | The slot. `:57652`. |
| `m_maxStackSize` | `1` | A worn accessory. |
| `m_maxQuality` | `1` | No upgrade tiers in v0.x (Lens precedent). |
| `m_weight` | ~1.0 | Cosmetic; tune to taste. |
| `m_teleportable` | `true` | No ore-ban interaction. |
| **`m_maxAdrenaline`** | **`80f`** (config) | **THE cap.** This is what makes the bar appear + sets ~80. `:57844`. |
| **`m_fullAdrenalineSE`** | **our SE asset** | **THE power.** Points at §3.3's StatusEffect. `:57846`. |
| `m_useDurability` | `false` | The *trinket* doesn't wear out (it's not a battery like the Lens). Also self-excludes it from its own repair set (§2.2). |

All registration gated behind the standalone mod's server/registrar context (the
`ServerContext.OnSBServer` / `Registrar` pattern). ObjectDB wiring (item + recipe)
mirrors `SunstoneLens.DoObjectDBWiring`.

### 3.3 The custom StatusEffect (the ONLY real custom logic)

Author one `StatusEffect` subclass — call it provisionally `SE_ForgeRepair`. Vanilla
SE subclasses follow a fixed shape (e.g. `SE_Burning :24740`, `SE_Finder :24986`,
all `: StatusEffect`). Build it as a `ScriptableObject` template registered into
ObjectDB so `m_fullAdrenalineSE` can reference it; vanilla **clones** it on apply
(`SEMan.AddStatusEffect → statusEffect.Clone()`, `:24397`), so the template is never
mutated.

The repair happens **on apply** (one-shot, §2.3 O2). Override `Setup(Character)`
(`:26456` base) — called once when the cloned SE is added — to do the work:

```csharp
public override void Setup(Character character) {
    base.Setup(character);                       // vanilla start-effects/message
    if (character is Player p && p == Player.m_localPlayer) {  // local-only; durability is client state
        foreach (var it in p.GetInventory().GetEquippedItems()) {   // :57192 public equipped set
            if (it?.m_shared == null || !it.m_shared.m_useDurability) continue;
            float max = it.GetMaxDurability();                       // :58130
            it.m_durability = Mathf.Min(it.m_durability + RepairAmount, max);  // +5 clamp
            // (O1b variant: pick argmin durability% instead of looping all)
        }
    }
}
```

Set the SE's `m_ttl` small/near-zero so it self-expires after doing its one-shot work
(it carries no ongoing buff). Optionally give it an icon + `$se_*` name/tooltip so the
HUD shows a brief "Forge-tempered" flash on cap-fire (cosmetic; not load-bearing).

> **Why `Setup` and not a Harmony patch:** vanilla *already* calls our SE for us at
> the cap (`:19506`). We don't need to patch `AddAdrenaline`, `UpdateEquipment`, or
> anything else. The clean-side surface is **one SE subclass + one item registration**
> — strictly smaller than the Lens (which patched `DrainEquipedItemDurability`). No
> third-party code touched; only base-game members referenced (ADR-0001 clean).

### 3.4 Recipe / tier / station (provisional — Daniel tunes)

| Aspect | Proposal | Rationale |

| Station | a forge-family station | Thematic (forge master). Could be vanilla `forge` or an SBPR bench — Daniel's call when the standalone mod's station story is set. |

### 3.5 Icon plan (SpecCheck hard requirement — the Kit icon lesson, t_ee1e763b)

A real icon is a **hard requirement**: the crafting UI indexes `m_icons[0]` and a
null icon throws client-side (the Cartographer's Kit icon-crash lesson). Follow the
Lens pattern exactly:
- Ship a PNG in the standalone mod's `assets/icons/items/` (e.g.
  `forgemaster_trinket_v0.1.png`), loaded via the `LoadPngAsSprite` helper.
- `ConstructItemShell` pre-seeds a **magenta fallback** so a missing PNG degrades to
  "ugly, never crash."
- The standalone mod's **own** SpecCheck C1-equivalent boot check must assert
  `m_icons[0]` is the real loaded sprite (not the fallback) and scream if the PNG
  didn't ship. v0.1 placeholder art is fine; the check guards the crash, not the art
  quality.

## 4. Repo / project structure proposal (Daniel ratifies at merge)

The card asks: reconcile "standalone mod" against the **locked HYBRID repo model**
(2026-06-15: the SBPR monorepo is the *release surface*; WorldZones lives in its own
repo as a pinned substrate dependency). "Standalone mod" can mean two materially
different things — and the distinction is the decision Daniel makes here.

### 4.1 The repo facts this must reconcile with (grounded)

- The monorepo today ships **one** plugin: `src/SBPR.Trailborne/` →
  `SBPR.Trailborne.dll` (`AssemblyName`/`RootNamespace` = `SBPR.Trailborne`,
  `<Version>0.2.25</Version>`).
- There is **no `.sln`**; CI builds the **one csproj by explicit path**
  (`ci.yml:120`, `release.yml:131`).
- `Directory.Build.props` resolves the Valheim/BepInEx reference paths **repo-wide**
  (shared by any project under the tree).
- `scripts/pack-modpack.sh` packs **one** plugin folder
  (`BepInEx/plugins/SBPR.Trailborne/` + that DLL + icons, `:14`, `:95-97`); the DLL
  path is a `--dll` arg, defaulting to the Trailborne build output.
- The hybrid model precedent (WorldZones) is the *"own repo, pinned dependency"*
  shape — used because WorldZones is a **substrate** other mods consume.

### 4.2 Option A — genuinely separate repo, released independently

A third GitHub repo (e.g. `SBPRForgeMaster`), its own BepInEx plugin, its own CI,
its own Thunderstore release.

- **Pros:** maximal independence; can version/release on its own cadence; mirrors the
  WorldZones precedent literally.
- **Cons:** this item is **not a substrate** — nothing else depends on it (unlike
  WorldZones). A third repo means duplicating the whole build/CI/pack/SDK-resolution
  scaffold (`Directory.Build.props`, `scripts/setup.sh`, `fetch-sdk.sh`, the CI server-
  asset fetch) for a single small trinket. High overhead, low payoff. The hybrid model
  put WorldZones in its own repo **because it's a dependency**, not as a default for
  every mod.

### 4.3 Option B — new plugin PROJECT in the monorepo, ships as its own DLL ⭐ recommended

Add `src/SBPR.ForgeMaster/SBPR.ForgeMaster.csproj` →
`SBPR.ForgeMaster.dll` (its own `BepInPlugin` GUID, its own `<Version>`, its own
`Runtime/SpecCheck.cs`). It is a **standalone plugin** — a separate DLL, separately
loadable, NOT a feature folder inside Trailborne — but it lives in the monorepo and
reuses the shared build scaffold.

- **Pros:**
  - **Honors "standalone mod"** in the way that matters to gameplay/release: its own
    plugin, own GUID, own version, own SpecCheck manifest, independently
    enable/disable-able. A server can run Trailborne without it or vice-versa.
  - **Reuses** `Directory.Build.props` SDK resolution, CI server-asset fetch, the
    pack script, the test harness — zero duplication.
  - Matches the hybrid model's *intent*: the monorepo is the **release surface**;
    WorldZones is separate **only because it's a consumed substrate**. ForgeMaster is
    a **leaf** (nothing depends on it), so it belongs on the release surface, not in a
    third repo.
  - Keeps the no-map thesis clean: it's not *in* Trailborne, so it doesn't dilute
    Trailborne's identity, but it doesn't pay the third-repo tax either.
- **Cons / required scaffold work (the impl card's job, not this card's):**
  - CI builds one csproj by path today → add a second build+test step (or introduce a
    `.sln` / `dotnet build` glob). Small, mechanical.
  - `pack-modpack.sh` packs one plugin dir → either pack a second plugin folder or
    parameterize. Small.
  - A second `BepInPlugin` GUID + version line.

### 4.4 Recommendation

**Option B (in-monorepo standalone plugin project).** Rationale: the hybrid model
splits a repo out **when it's a substrate dependency** (WorldZones). The Forge
Master's Trinket is a **content leaf** — nothing consumes it — so the cost/benefit of
a third repo is all cost. Option B delivers everything "standalone mod" means for
gameplay and release (own DLL, GUID, version, SpecCheck, on/off) while reusing the
build/CI/pack scaffold the monorepo already maintains. If Daniel later wants it on
Thunderstore as a separate listing, Option B still supports that (pack a second
plugin folder) without a repo split.

**If Daniel picks A**, the scaffold-card scope grows to "stand up a new repo with the
full build/CI/pack/SDK story" before any gameplay code. **If B**, the scaffold-card
adds one project + extends CI/pack. Either way, **repo/project creation is a separate
Daniel-gated step** (Out of scope here) that happens only after this structure call is
ratified.
